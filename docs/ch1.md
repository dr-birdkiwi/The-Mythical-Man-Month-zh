# 第 1 章 焦油坑（The Tar Pit）

岸上的船儿，如同海上的灯塔，无法移动。

——荷兰谚语

Een schip op het strand is een baken in zee.

史前史中，没有任何景象比巨兽在焦油坑中垂死挣扎更令人震撼。我们仿佛能看到恐龙、猛犸象、剑齿虎在焦油中拼死挣扎的身影。它们越是猛烈地挣扎，焦油便缠得越紧；没有任何猛兽能凭借蛮力或技巧挣脱束缚，它们最终都沉入了坑底。

过去几十年间的大型系统开发，就如同这样一个焦油坑。许多庞大而强健的动物在其中剧烈挣扎。他们中大多数开发出了能运行的系统——然而，只有极少数项目真正达到了目标、进度和预算的全部要求。各种团队——无论规模大小、配置精简或是庞杂——一个接一个地陷入其中。表面上看，没有哪一个单独的问题足以致命，每个问题单独来看似乎都能解决；然而当它们交织叠加在一起时，团队的前进便越来越艰难。问题的棘手程度总是超乎所有人的预料，而困难的根源又难以看清。然而，若想解决问题，就必须先尝试去理解它。

因此，就让我们先来认识一下软件开发这门行业，以及这门行业中的乐趣与苦恼。

## 编程系统产品

报纸上时常会出现这样的报道：两个程序员在一间简陋的车库里，编写出了超越大型团队产出的重要程序。每个程序员都倾向于相信这类神话，因为他知道自己能以远超产业化团队（1000 行代码/年）的生产率，独立开发任何程序。

但为什么产业化的开发团队没有被这种专注的二人组合所取代呢？我们必须看清楚：两者产出的是完全不同的东西。

![编程系统产品的演进](figures/fg1-1.jpg)
> 图 1.1：编程系统产品的演进

图 1.1 左上角是**程序（Program）**。它本身是完整的，能由作者在所开发的系统平台上运行。它通常是车库里产出的成果，也是衡量单个程序员生产率的基准。

有两条路径可以将程序转化为更有价值（但成本也更高）的东西，对应图中的两条边界。

**水平边界以下**，程序变成**编程产品（Programming Product）**。这是任何人都可以运行、测试、修复和扩展的程序。它能在多种操作系统平台上运行，支持多套数据。要成为通用的编程产品，程序必须按照大家认可的规范编写；特别是输入的范围和形式必须加以扩展，以覆盖所有可以合理使用的基本算法。程序还必须经过彻底测试，确保其稳定可靠、值得信赖——这意味着需要准备、运行并记录完整的测试用例库，以检查输入的边界条件和各种取值范围。此外，还必须有完备的文档，使任何人都能使用、修复和扩展它。经验数据表明，相同功能的编程产品，其开发成本至少是一个经过测试的程序的三倍。

**垂直边界右侧**，程序变成**编程系统（Programming System）**中的一个构件。编程系统是一组在功能上能相互协作的程序集合，具有规范的接口格式，各部分可以交互，并能组装成完整的系统。要成为系统构件，程序必须满足严格的要求：输入和输出在语法和语义上都必须符合精确定义的接口规范；同时还要遵守预先规定的资源限制——内存、I/O 设备、CPU 时间。最后，程序必须与其他所有系统构件一起，以任意可能的组合方式进行测试。由于组合数量随构件增加而急剧增长，测试范围极其广泛。各种意料之外的交互又会产生难以察觉的 bug，使测试工作极为耗时，因此相同功能的编程系统构件，其成本至少是独立程序的三倍。若系统组件很多，成本还会更高。

图 1.1 右下角是**编程系统产品（Programming Systems Product）**。它与以上所有情况都不同——成本高达九倍。然而，这才是真正有用的产品，也是大多数系统开发的最终目标。

## 职业的乐趣

编程为什么有趣？从业者希望从中获得怎样的回报与快乐？

首先，是纯粹的创造之乐。就像孩子喜欢玩泥巴，成年人也喜欢创造事物，尤其是亲自设计的事物。我想，这种快乐是上帝造物之喜悦的某种映照——那种在每一片独特的树叶、每一朵崭新的雪花上都能感受到的喜悦。

其次，快乐来自于创造出对他人有用的东西。我们内心深处渴望自己的劳动成果能被他人使用，并切实帮助到他们。从这个角度看，这与孩子用黏土为"爸爸的办公室"捏一个铅笔筒，并无本质区别。

第三，快乐来自于那种近乎魔术的力量——将相互咬合的零部件拼装在一起，看着它们精妙地运转，得到预先期望的结果。可编程的计算机，其魅力不亚于弹珠台或点唱机。

第四，是学习的乐趣，源自这项工作永不重复的特性。人们面临的问题，总会在某些方面与以前不同。解决问题的过程因此充满新知：有时是实践层面的，有时是理论层面的，通常二者兼而有之。

最后，乐趣还来自于工作在这样一种极易驾驭的介质上。程序员就像诗人一样，几乎纯粹工作于思维之中，凭借想象力凭空构建自己的"城堡"。很少有哪种介质如此灵活，如此易于精炼和重建，如此容易将脑中的构想付诸实现。（不过我们将会看到，这种易于驾驭的特性也有它自己的问题。）而程序终究与诗歌不同——它是实实在在的东西，能够运行，能独立产生可见的输出：打印结果、绘制图形、发出声音、驱动机械。神话传说中的魔法，在我们这个时代已成为现实。只要在键盘上敲入正确的咒语，屏幕便会活动变幻，呈现出前所未有的或确实存在的事物。

编程之所以有趣，在于它满足了我们内心深处对创造的渴望，也愉悦了每个人心中那份原始的情感。

## 职业的苦恼

然而这个过程并非全是喜悦。若能事先了解编程固有的一些烦恼，当它们真的出现时，我们便能更加坦然地面对。

首先，是必须追求完美。计算机变戏法的方式如出一辙：若咒语中有一个字符、一处停顿与正确形式不符，魔术便不会发生。（在现实生活中，极少有哪种活动要求完美，人类本来就不擅长此道。）我认为，学习编程最难的部分，正是在心态上向追求完美靠拢。

其次，目标由他人设定，资源由他人提供，信息也由他人掌控。程序员很少能掌控自己的工作环境和工作目标。用管理学的术语说，个人的权威与所承担的责任往往不相匹配。不过，几乎在所有领域，正式授权都很难与责任完全对等；而实际上的（而非名义上的）权威，往往来自一次次任务的圆满完成。

对系统程序员而言，依赖他人尤为痛苦。他必须依赖他人的程序，而这些程序往往设计欠妥、实现粗糙、发布不完整（缺少源代码或测试用例），或者文档一塌糊涂。因此，系统程序员不得不花费大量时间去研究和修补那些本该可靠完整的东西。

下一个烦恼：概念设计固然有趣，但追查琐碎的 bug 不过是一种重复性的苦差。创造性活动与枯燥沉闷的劳动如影随形，程序编制工作也不例外。

此外，调试和查错的收敛速度往往只是线性的，甚至更差——呈平方级的复杂度。结果，测试一拖再拖，找到最后一个 bug 所花费的时间，远比找到第一个要多。

最后一个苦恼，有时也是一种无奈——投入了大量心血，产品在即将完成甚至终于完成之时，却已显得陈旧过时。也许同事或竞争对手已在追逐更新更好的方案；也许替代品不仅停留在构想阶段，而且已经在开发之中了。

现实情况通常比上面说的要好一些。当你的产品开发完成时，那个"更优秀的新产品"往往还无法投入使用，不过是停留在大家的谈论之中。而且它同样需要数月的开发时间。说到底，只有当实际需要时，最新的设想才会付诸实施——因为已有的系统已经能满足要求，投资已经得到了回报。

诚然，产品开发所依赖的技术在不断进步，设计一旦冻结，从概念上说便开始落伍了。不过，实际产品需要一步一步按阶段实现。评价一个实现是否落后，应当以其他已有的系统为参照，而非以尚未实现的构想。因此，我们所面临的挑战，是在现有的时间和资源约束下，找到解决实际问题的切实可行之道。

这，就是编程。一个令许多人在其中挣扎的焦油坑，也是一种乐趣与苦恼并存的创造性活动。

对许多人而言，其中的乐趣远大于苦恼。本书其余部分将尝试搭建几座桥梁，为穿越这片焦油坑提供一些指引。
